# 战绩录入与比赛详情设计草案

## 目标

本模块面向英雄联盟国服联赛战绩录入。目标是支持管理员把已完成比赛录入系统，并生成可公开查看的历史比赛详情，包括赛程信息、小局结果、出场名单、战绩数据、回放文件、积分结算、P 币流水和身价变化。

当前不应假设存在可公开调用、稳定合规的腾讯国服 Web API。自动导入能力应设计为“管理员本机采集工具 + 自建后端入库 + 网站只读展示”，而不是网站服务器直接访问国服官方接口。

## 推荐技术路线

### 管理员本机采集工具

管理员在已登录 LOL 国服客户端的设备上运行本地采集工具。工具读取本机 LCU 端口和 token 后，通过本地客户端接口按 `gameId` 采集对局数据。

LCU 接口体系中存在 match-history、ACS 和 replays 相关接口记录，例如：

- `/lol-match-history/v1/games/{gameId}`
- `/lol-match-history/v1/game-timelines/{gameId}`
- `/lol-acs/v1/games/{gameId}`
- `/lol-acs/v1/gametimelines/{gameId}`
- `/lol-replays/v1/rofls/{gameId}/download`
- `/lol-replays/v1/rofls/{gameId}/watch`

这些接口只能作为管理员本机采集能力使用。实现时不应上传、保存或暴露 LCU token，也不应让公网网站服务器直接访问管理员客户端。

### 数据入库链路

建议链路如下：

1. 管理员登录 LOL 国服客户端。
2. 管理员启动本地采集工具。
3. 管理员输入 `gameId`。
4. 工具检查客户端登录状态和对局可访问性。
5. 工具拉取对局详情、时间线和 ACS 数据。
6. 工具尝试下载 `.rofl` 回放文件。
7. 工具解析结构化战绩数据。
8. 管理员确认该 `gameId` 对应的赛季、轮次、比赛和小局。
9. 管理员补充主观表现分、特殊裁定、奖励或罚款。
10. 工具把解析后的数据和必要附件上传到自建后端。
11. 网站从自建后端读取公开比赛详情。

自动采集成功后也不应直接生效，仍需要管理员确认并发布结算。

## 人工兜底

LCU 自动采集可能因为权限、区服、近期战绩可访问性、补丁版本、回放过期、接口变动或合规风险失败。因此系统必须提供人工录入路径。

人工录入至少需要支持：

- 手动录入大比分和每小局胜负。
- 手动录入双方每小局出场 5 人和位置。
- 标记租借、替补、跨位置等情况。
- 上传战绩详情截图。
- 上传或补充回放文件。
- 填写管理员裁定说明。

人工录入和自动导入应进入同一套结算流程，避免两套数据模型。

## 赛程与比赛状态

第一赛季赛程按轮次作为主标签，日期作为副标签。历史比赛详情页建议采用左侧比赛列表、右侧详情的布局，左侧可按赛季、轮次、队伍、状态和赛制筛选。

公开比赛状态建议包括：

- `scheduled`：已排期。
- `upcoming`：即将进行，可由比赛时间推导。
- `live`：正在进行，可附带直播间链接或直播备注。
- `finished`：已完结，正常结算。
- `forfeit`：弃赛，按弃赛规则结算。
- `postponed`：延期。
- `cancelled`：取消，可选。

`finished` 和 `forfeit` 都会影响积分、P 币和可能的赛事记录，但应按不同规则结算。第一版不需要公开 pending 状态；后台可保留草稿态，但未发布内容不对外展示。

## 小局数据

每场 BO2/BO3 比赛应按小局记录详情。每小局至少包含：

- 小局序号。
- 胜方。
- 蓝方和红方。
- 双方出场 5 人。
- 每名选手的位置。
- 每名选手的站内选手 ID 和游戏昵称映射。
- 是否租借。
- 原队伍和本场代表队伍。
- 英雄。
- KDA。
- 补刀、经济、输出、承伤、参团率等统计数据。
- 小局截图或回放附件，可选。

由于 BO3 中可能换人或租借，阵容应按小局保存，而不是只按整场比赛保存一份。

## 结算范围

系统应自动计算：

- 大比分。
- 小局胜负。
- 积分变化。
- 奢侈税。
- 租借费。
- P 币流水。
- 客观身价变化。

管理员需要手动录入或确认：

- 主观表现分。
- 主观表现分原因。
- 罚款。
- 竞猜奖励。
- 特殊裁定。
- 结果修正说明。

积分榜应最终由已发布的比赛结果自动生成。管理员可以修正错误录入，但修正应走审计和重新结算流程。

## P 币流水

队伍当前 P 币不应作为唯一手填数值长期维护。所有 P 币变化应保存为流水，余额由流水汇总生成。

建议流水类型包括：

- `match_reward`：比赛奖励。
- `luxury_tax`：奢侈税扣款。
- `loan_fee`：租借费。
- `forfeit_penalty`：弃赛罚款。
- `quiz_reward`：竞猜奖励。
- `admin_adjustment`：管理员手动调整。
- `catchup_fund`：追赶基金。
- `league_reclaim`：联盟回收。

每条流水应记录来源比赛、操作人、原因、创建时间和是否由某次结算版本生成。

## 身价变化

身价变化由客观分和主观表现分组成。

- 客观分由函数计算，输入应来自胜负、选手身价、对位和战绩等客观因素。
- 主观表现分由管理员录入，并记录原因。
- 最终变化 = 客观变化 + 主观表现分。

公开比赛详情中建议展示：

- 赛前身价。
- 客观变化。
- 主观表现分。
- 主观原因。
- 最终变化。
- 赛后身价。
- 操作管理员。
- 结算时间。

具体客观分公式和主观表现分范围应由规则文档解释，并接受观众监督。

## 权限

第一阶段权限可以简化为：

- 公开用户：包含游客和选手，只能查看公开赛程、结果和比赛详情。
- 队长：可参与本队相关确认或申诉，具体能力后续再定。
- 管理员：录入比赛、导入数据、发布结算、上传附件、添加主观分和特殊流水。
- 超级管理员：管理登录权限、回滚或撤销重大错误、修改规则版本。

管理员、队长和超级管理员需要登录。公开数据面向所有人。

## 审计与修正

建议采用简单的版本化发布方案，而不是直接覆盖旧结果。

1. 每场比赛第一次发布为 `v1`。
2. 管理员修改已发布结果时，生成 `v2`。
3. 旧版本标记为 `voided`，但不删除。
4. 系统生成反向流水，抵消旧版本造成的积分、P 币和身价变化。
5. 系统基于新版本重新生成积分、P 币和身价变化。
6. 每次修改必须填写原因。
7. 比赛详情页可显示当前生效版本，并在管理员视图中查看历史版本。

后台第一版只需要两个核心操作：

- 发布结果。
- 撤销并重新结算。

这个方案可以保证比赛结果、积分、P 币和身价变化有据可查，同时避免一开始引入复杂审批流。

## 风险与约束

- LCU 接口不是公开网站 API，可能随客户端版本变动。
- 本地工具只能访问已登录客户端有权限查看的对局。
- 回放文件可能受补丁版本、过期策略和客户端机制影响。
- `.rofl` 文件下载成功后应上传到自建存储，并记录游戏版本。
- 页面应提示回放可能仅支持对应版本客户端播放。
- 不应保存或上传 LCU token。
- 不应绕过客户端权限或访问管理员无权限查看的数据。
- 自动导入失败时，人工录入流程必须可用。

详细战绩字段、账号映射、回放策略和身价客观分输入字段需要等待 LCU 采集工具的真实输出后再锁定，待细化清单见 [战绩详情待细化项](MATCH_DETAIL_OPEN_ITEMS.md)。
