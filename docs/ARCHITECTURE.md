# 架构说明

## 阶段定位

当前阶段是“静态官网 + 模块化业务基础”。它不引入构建工具，也不接数据库，目标是先把原型页面整理成可以用 Git 长期维护的结构。由于源码使用浏览器 ES Module，本地预览建议通过静态 HTTP 服务访问。

后续如果要升级为完整联赛管理系统，可以在现有边界上迁移：

- `src/data` 替换为 API 数据源。
- `src/services` 保留为前端共享计算规则，或抽到后端共用包。
- `src/features` 拆成框架组件，例如 React/Vue/Svelte 组件。
- `assets` 继续作为公开静态资源目录。

## 模块边界

### `index.html`

页面壳层，只保留语义结构和挂载节点。它不承载联赛数据，也不写业务计算。

### `src/data`

当前联赛数据源，包括：

- 公告
- 队伍、选手、P 币、积分、排名
- 规则条目
- 赛程
- 首页统计项

新增或修改联赛内容时，优先在这里改。

### `src/services`

纯业务计算，不直接操作 DOM。当前包括：

- P 币格式化
- 队伍总身价计算
- 工资帽参考线计算
- 奢侈税分段计算
- 租借费分配
- P 币兑换次数计算

这个目录里的函数应该保持可测试、可复用，后续适合补单元测试。

### `src/features`

页面功能模块，负责把数据渲染到 DOM，并绑定交互事件。当前拆分为：

- `content.js`：公告、规则、赛程、首页统计。
- `standings.js`：积分榜。
- `teams.js`：队伍卡片和搜索。
- `calculators.js`：三个计算器。
- `navigation.js`：移动导航、规则折叠和滚动高亮。

### `src/styles`

样式入口。当前保留原视觉方向：深色电竞风、霓虹边框、队伍徽记光效和响应式布局。

## 后续系统边界建议

完整联赛管理系统建议拆成四个域：

- 官网展示域：公告、赛程、积分榜、队伍、选手资料。
- 赛事运营域：比赛录入、赛果确认、积分结算、弃赛处理。
- 经济系统域：P 币流水、工资帽、奢侈税、租借费、追赶基金。
- 管理后台域：登录权限、队长/管理员角色、数据审计、发布流程。

在后端接入前，前端不要把“录入”和“结算”逻辑散落在 UI 事件里；计算规则应继续沉到 `services` 或未来的共享业务包。

战绩录入不应设计成公网网站服务器直接访问国服官方接口。国服对局自动导入应通过管理员本机采集工具读取已登录 LOL 客户端的 LCU 接口，解析后上传到自建后端，再由网站展示公开比赛详情。具体实现思路见 [战绩录入与比赛详情设计草案](MATCH_RESULT_IMPORT.md)。
