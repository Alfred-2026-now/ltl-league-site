# 战绩详情待细化项

本文档记录战绩详情模块中暂不锁死的数据结构和规则。原因是这些内容依赖后续管理员本机 LCU 采集工具的真实输出，以及联赛规则文档中对身价客观分的最终解释。

## LCU 采集工具输出

待确认内容：

- 采集工具输出的原始 JSON 字段结构。
- match-history、ACS、timeline 三类数据之间的字段对应关系。
- 是否能稳定拿到每名玩家的英雄、位置、KDA、经济、补刀、输出、承伤、参团率等字段。
- 是否能稳定识别蓝方、红方、胜方和小局时长。
- 是否能从 `gameId` 直接获取足够信息来区分 BO2/BO3 中的第几小局。
- 工具版本、客户端版本、游戏版本等元数据字段。

开发要求：

- 网站和后端不应直接依赖 LCU 原始字段。
- 应先保存原始响应，再转换成站内标准比赛数据。
- 转换器应允许随 LCU 字段变化单独升级。

## 小局详细战绩字段

待确认内容：

- 选手战绩字段最终集合。
- 团队战绩字段最终集合。
- 时间线事件是否需要在页面展示。
- 是否展示英雄、符文、召唤师技能、装备、地图资源、击杀事件等细节。
- 是否需要 MVP、关键团战、关键事件等人工标注字段。

开发要求：

- 第一版页面只锁定小局、胜方、阵容、位置、租借和基础结算。
- 详细战绩区域需要预留扩展容器。
- 真实字段应等采集工具产出稳定样本后再定。
- 在 LCU 结构化数据未完成前，小局可通过 `scoreScreenshots` 上传战绩截图作为兜底。
- 若同一小局同时存在截图和结构化数据，详情页默认显示截图模式，并允许切换到详细图表模式。
- 若只有截图没有结构化数据，只显示截图和基础小局信息。
- 若截图和结构化数据都不存在，只显示基础比分、胜方和“详细战绩待录入”提示。

## 账号到站内选手映射

待确认内容：

- 国服账号、召唤师名、游戏名、tag、PUUID 或其他标识之间的可用关系。
- 选手改名后的历史匹配策略。
- 租借、替补、跨位置出场时由谁确认映射。
- 队长是否拥有本队映射确认权限。

开发要求：

- 不应把账号直接等同于站内选手。
- 每小局阵容需要保存“账号标识”和“站内选手 ID”的映射结果。
- 映射需要支持管理员或队长人工确认。
- 租借时需要记录原队伍和本场代表队伍。

## 回放与附件

待确认内容：

- `.rofl` 文件下载成功率和过期策略。
- 回放文件保存位置和公开下载权限。
- 回放文件大小限制。
- 对应客户端版本提示文案。
- 战绩截图、裁定说明、人工上传附件的格式和大小限制。

开发要求：

- 回放文件应上传到自建存储，不依赖客户端长期可下载。
- 页面需要显示回放版本和可用性提示。
- 自动回放下载失败时，人工附件流程必须可用。

## 身价客观分输入字段

待确认内容：

- 客观分公式最终需要哪些战绩字段。
- 对位表现如何识别。
- 不同位置是否使用不同权重。
- 胜负、身价差、KDA、参团率、输出占比、承伤等字段的权重。
- BO2/BO3、弃赛、租借和替补是否影响客观分。

开发要求：

- 客观分函数应保持纯函数，便于测试。
- 主观表现分由管理员录入，并记录原因。
- 最终身价变化应展示客观变化、主观变化和最终变化。
- 规则未定前，页面只展示预留字段或示例静态结算。

## 审计和修正细节

待确认内容：

- 普通用户是否能查看历史作废版本。
- 队长是否能看到本队相关修正记录。
- 反向流水和新流水在公开页面中的展示粒度。
- 已发放竞猜奖励在比赛修正后的处理规则。

开发要求：

- 已发布比赛结果不直接覆盖。
- 修改已发布结果时生成新版本。
- 旧版本保留并标记作废。
- 修改必须填写原因。
